### [LAB-01] 실험 결과 데이터 (10만 건 기준)

| Batch Size | JDBC (ms) | JPA (ms) | 차이 (Ratio) |
| --- | --- | --- | --- |
| **10** | 4,256 | 6,564 | **1.54x** |
| **100** | 2,260 | 2,651 | **1.17x** |
| **500** | **1,975** | **2,183** | **1.11x** (Best) |
| **1,000** | 1,931 | 2,218 | 1.15x |
| **2,000** | 1,966 | 2,236 | 1.14x |
| **5,000** | 1,916 | 2,285 | 1.19x |

### 핵심 요약 및 결론

1. **배치 사이즈 500이 '최적점'**: 10에서 500까지는 성능이 비약적으로 좋아지지만, 그 이상(1,000~)부터는 속도 향상이 거의 멈춥니다. (한계 효용 체감)
2. **JPA도 최적화하면 충분히 빠름**: 최적 배치 구간(500~1000)에서 JPA는 JDBC와 단 **11~15%** 밖에 차이가 나지 않습니다. 특수한 경우가 아니라면 JPA 배치를 써도 충분합니다.
3. **네트워크 비용이 범인**: 배치 사이즈가 작을 때(10) 느린 이유는 데이터 양보다 DB와 통신을 너무 자주(왕복) 하기 때문입니다.
